# 🦉 รายงานสรุปความรู้เรื่อง Git Repository (Repo)

รายงานฉบับนี้สรุปความรู้และความเข้าใจเกี่ยวกับ **Repository (Repo)** รวมถึงปัญหาที่พบในการทำงานร่วมกับน้องๆ ใน ChZ Agent Corp เพื่อให้แกใช้เป็นคู่มืออ้างอิงและป้องกันไม่ให้เกิดปัญหาขึ้นอีกในอนาคตนะแก

---

## 1. Repo คืออะไร?

**Repo (ย่อมาจาก Repository)** คือ **"คลังเก็บไฟล์งาน"** ที่อยู่ภายใต้การดูแลของระบบ **Git** 
* เปรียบเหมือนตู้เอกสารอัจฉริยะที่ไม่ได้เก็บแค่ไฟล์เวอร์ชันล่าสุด แต่จะจดจำประวัติศาสตร์การแก้ไขทุกอย่างย้อนหลังตั้งแต่เริ่มสร้างตึก (Commit History)
* ทุกครั้งที่มีการแก้ไข เพิ่ม หรือลบโค้ด แล้วสั่ง `git commit` ระบบจะบันทึกการเปลี่ยนแปลงนั้นไว้ในโฟลเดอร์ลับที่ชื่อว่า `.git` เสมอ

---

## 2. สิ่งที่ควรรู้เกี่ยวกับ Repo

* **Working Directory vs Git History (โฟลเดอร์ `.git`):**
  * **Working Directory:** คือโฟลเดอร์งานที่แกมองเห็นและเปิดดูได้ปกติ (เช่น ไฟล์โค้ด .py หรือเอกสาร .md)
  * **Git History (โฟลเดอร์ `.git`):** คือระบบฐานข้อมูลลับหลังบ้านที่คอยเก็บประวัติการบันทึกทั้งหมด มีขนาดที่แตกต่างกับโฟลเดอร์ข้างนอกได้ลิบลับ (อย่างที่ตึกเราบวม 36GB ก็เพราะประวัติหลังบ้านตัวนี้)
* **Tracked vs Untracked Files:**
  * **Tracked (ไฟล์ที่โดนติดตาม):** ไฟล์ที่ Git คอยจับตาดูและเซฟประวัติเมื่อมีการเปลี่ยนแปลง
  * **Untracked (ไฟล์ที่ไม่ได้ติดตาม):** ไฟล์ที่อยู่ในโฟลเดอร์งานเฉยๆ แต่ Git ไม่ได้สนใจ (รวมถึงไฟล์ที่เขียนระบุไว้ใน `.gitignore`) ซึ่งจะไม่ถูกนำไปบันทึกประวัติ หรือไม่ย้ายตามเมื่อใช้คำสั่งของ Git

---

## 3. Repo กับปัญหาที่เกิดขึ้นในการทำงานล่าสุด

ปัญหาล่าสุดที่ตึกเราแยกเป็นสองตึก (`WTJ_Content_Studio` กับ `Team_Content_Studio`) เกิดจากกลไกการทำงานของ Repo ดังนี้:

1. **การ Rename แบบครึ่งๆ กลางๆ (Git mv):**
   * ตอนที่เราเปลี่ยนชื่อโฟลเดอร์ใน Git ตัว Git จะย้ายไปเฉพาะไฟล์ที่ถูก **Tracked** เท่านั้น
   * แต่ไฟล์งานเก่าๆ บางส่วน หรือไฟล์มีเดียที่ไม่ได้โดน Tracked ยังคงตกค้างอยู่ที่เดิม ทำให้ระบบไฟล์จริงของ macOS ไม่ยอมลบโฟลเดอร์ตึกเก่าออก
2. **น้องๆ วิ่งกลับไปสร้างตึกเก่า (Hardcoded Path):**
   * น้องๆ (สคริปต์เอเจนต์) มีการระบุ Path ในโค้ดว่า `WTJ_Content_Studio` 
   * เมื่อสคริปต์รันทำงานอัตโนมัติ มันเลยสร้างโฟลเดอร์ชื่อเก่าขึ้นมาใหม่เพื่อบันทึกงานเรื่อยๆ จนกลายเป็นตึกแฝดหลอกหลอน

## 4. สาเหตุที่ทำให้ Repo บวม (Repository Bloat)

การที่ Repo มีขนาดใหญ่เกินไป (เช่น ทะลุ 10-30 GB) มักเกิดจาก 3 สาเหตุหลัก:

1. **การเผลอเอาไฟล์ใหญ่ (Binary Files) ใส่เข้า Git:**
   * ไฟล์ตระกูลมีเดีย เช่น VDO (`.mp4`, `.mov`), เสียง (`.wav`, `.m4a`), รูปภาพดิบ รวมถึงไฟล์บีบอัด (`.zip`) และฐานข้อมูล (`.sqlite3`)
2. **การลบไฟล์ออก แต่ไม่ลบประวัติ (History):**
   * การกด Delete ไฟล์ VDO ออกจากโฟลเดอร์ปกติ **ไม่ได้ช่วยให้ Repo เล็กลง** เพราะในระบบ `.git` หลังบ้านยังต้องเก็บไฟล์ VDO ตัวนั้นไว้ในอดีต (ประวัติ) เพื่อให้แกสามารถย้อนเวลากลับไปดึงข้อมูลได้ ขนาดของมันเลยยังสะสมอยู่ตลอดไป
3. **การลืมตั้งค่า `.gitignore`:**
   * ไม่มีตัวกรองไฟล์ขยะ ทำให้พวกโฟลเดอร์จำลองระบบอย่าง `venv/`, cache หรือ logs ต่างๆ หลุดเข้าไปสะสมใน Git

## 5. วิธีป้องกันและแก้ปัญหา Repo บวม

แนวทางปฏิบัติที่ดีที่สุด (Best Practices) เพื่อดูแลรักษา Repo ของเรามีดังนี้:

| วิธีการ | รายละเอียดแนวทาง |
|---|---|
| **1. ตั้งค่า `.gitignore` ตั้งแต่วันแรก** | ระบุประเภทไฟล์ที่ไม่ต้องการให้ Git ตรวจสอบ เช่น มีเดียไฟล์ (`*.mp4`, `*.mov`, `*.m4a`), โฟลเดอร์ `venv/`, cache, และ credentials ต่างๆ |
| **2. แยกเก็บไฟล์ใหญ่ไว้ข้างนอก** | สำหรับไฟล์ VDO ที่น้องๆ ต้องนำไปถอดเสียงหรือเตรียมอัปโหลด YouTube ให้เก็บไว้ในโฟลเดอร์ Local บนเครื่องที่อยู่นอกโปรเจกต์ หรือใส่ใน Cloud Storage แล้วดึงผ่าน Local Path แทนการลากเข้ามาใน Git |
| **3. ล้างประวัติย้อนหลังด้วย `git-filter-repo`** | หากมีไฟล์บวมหลุดเข้าไปในประวัติ ให้ใช้เครื่องมือกรองและลบไฟล์ประเภทนั้นๆ ออกจากประวัติ Git ทุก Commits ย้อนหลัง เพื่อลดขนาดโฟลเดอร์ `.git` จริง |
| **4. หมั่นกวาดขยะด้วย `git gc`** | ใช้คำสั่ง `git gc --aggressive --prune=now` เพื่อบีบอัดฐานข้อมูลและเคลียร์ขยะสะสมหลังทำการคลีนประวัติ |
