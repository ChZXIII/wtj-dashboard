## 🛠️ คู่มือความประพฤติและสกิลผู้เชี่ยวชาญ (Expert Guidelines)

ในการเขียนโค้ด การรีวิวงาน หรือการแก้บั๊ก คุณต้องปฏิบัติตามหลักการต่อไปนี้อย่างเคร่งครัด:

### 🦉 1. สกิลหลีกเลี่ยงข้อผิดพลาด (Karpathy Guidelines)
- **Think Before Coding:** อย่าเพิ่งเขียนโค้ดทันทีเมื่อพบความคลุมเครือ ให้ลิสต์ข้อสมมติฐานและแนวคิดที่จะใช้คุยตกลงกันก่อน
- **Simplicity First:** เขียนโค้ดให้น้อยที่สุดที่แก้ปัญหาได้จริง ไม่ต้องสร้าง Abstraction สปอยล์อนาคต (No speculative code)
- **Surgical Changes:** แก้ไขแบบเจาะจงเฉพาะบรรทัดที่จำเป็น ห้ามไปแก้ฟอร์แมตหรือส่วนอื่นที่ไม่ได้มีปัญหา รักษาสไตล์ดั้งเดิมของเจ้าของโค้ดเก่า
- **Goal-Driven Execution:** เปลี่ยนปัญหาให้เป็นเป้าหมายความสำเร็จที่เป็นรูปธรรม (เช่น รันผ่านเทสเคสอะไรบ้าง) และรันตรวจสอบจนแน่ใจ

### 🔍 2. สกิลวิเคราะห์และรีวิวระดับเจาะลึก (Scrutinize)
เมื่อต้องทำการตรวจสอบแผนงาน, ตรวจสคริปต์ หรือตรวจสอบ Pull Request (PR):
- **Outsider Stance:** มองงานชิ้นนั้นเสมือนคนนอกที่ไม่รู้จักคนเขียนมาก่อน เพื่อตัดอคติและความคุ้นชิน
- **End-to-End Tracing:** ไล่ตามทางเดินข้อมูล (Code Path) จาก Entry point ➡️ ฟังก์ชัน ➡️ ตัวแปร ➡️ ปลายทาง รวมถึงโค้ดที่ไม่ถูกแก้สองข้างทาง ไม่ตรวจทานเฉพาะช่วง Diff
- **Edge Case Focus:** หาจุดเปราะบางเมื่อเกิดข้อมูลว่าง (Null/Empty), ตัวอักษรพิเศษ (Unicode), ข้อมูลขนาดใหญ่, หรือการเรียกใช้พร้อมๆ กัน (Concurrency)
- **Report Format:** 
  - **Finding:** ระบุไฟล์และบรรทัดที่พบจุดอ่อนให้ชัดเจน
  - **Why it matters:** ผลลัพธ์เสียหายที่จะเกิดขึ้นจริง (ไม่ใช่แค่หลักการทฤษฎี)
  - **Evidence:** ข้อมูลชี้ชัดจากการไล่โค้ด
  - **Suggested Change:** ข้อเสนอโค้ดแก้ไขแบบมินิมอลเฉพาะจุด
  - ปิดท้ายรายงานด้วย **Verdict** เสมอ เช่น: `Verdict: ship` / `Verdict: fix-then-ship` / `Verdict: reject` พร้อมสรุปเหตุผลบรรทัดเดียว
