## 📝 กฎเหล็กประจำตัวของผม (Coding Principles)
- **Zero Trash Code:** เขียนโค้ดกระชับ คลีน และมีคอมเมนต์อธิบาย Logic ชัดเจนเพื่อไม่ให้ทีมงานสับสน
- **Strict Error Handling:** สคริปต์ระบบออโต้รันทุกตัวต้องเขียนระบบดักจับ Error และส่งแจ้งเตือนมาให้พี่เฟิสทราบเสมอ ไม่ปล่อยให้ระบบล่มเงียบหลังบ้าน
- **Security First:** ไฟล์ API Token หรือความลับ/สถานะต่างๆ (เช่น `token.json`, `token_sheets.json`, `client_secret.json`, `token_youtube.json`, `token_tiktok.json.bak`, `trade_sync_state.json`) **ต้องเก็บรวมไว้ในโฟลเดอร์ `credentials/` ที่รูทของโปรเจกต์เท่านั้น** ห้ามเอาออกมาวางเกะกะนอกตึก (รูทโฟลเดอร์) เด็ดขาด และสคริปต์ทุกตัวที่เขียนต้องชี้พาธไปอ่านที่โฟลเดอร์นี้เสมอ
- **No Pre-mature Implementation (ห้ามข้ามขั้นตอนคิวเขียนโค้ดก่อนได้รับการอนุมัติ):** ห้ามคิวชิงลงมือแก้ไขหรือเขียนโค้ดจริงจนกว่าพี่เก่งจะอนุมัติแผนงาน (Implementation Plan) ในแชทอย่างเป็นทางการเด็ดขาด ทุกๆ งานต้องเริ่มจาก Research -> เสนอแผน -> รออนุมัติ -> ลงมือทำตามลำดับเสมอ
- **System-Wide Dependency Verification (เช็กความเชื่อมโยงต้นทาง-ปลายทาง):** ทุกครั้งที่เขียนสคริปต์จัดการไฟล์ ซิงค์ข้อมูล หรือดาวน์โหลด ห้ามด่วนเดาพาธหรือใช้วิธีฮาร์ดโค้ดค่าเดียวทันทีเด็ดขาด ต้องตรวจสอบฝั่งหน้าบ้าน/แหล่งกำเนิดข้อมูลเสมอว่าระบบสามารถเลือกประเภทโปรเจกต์หรือโปรแกรมได้หรือไม่ เพื่อดึงฟิลด์ข้อมูลที่ระบุประเภทนั้นมาตรวจสอบเงื่อนไขและจัดการแยกแยะปลายทางของแต่ละโปรเจกต์ให้ถูกต้องร้อยเปอร์เซ็นต์
- **End-to-End Verification Only (ห้ามทึกทักเอาเองว่าระบบเสร็จโดยไม่ทำทดสอบจริงต้นน้ำยันปลายน้ำ):** ห้ามสรุปหรือแจ้งจบงานเด็ดขาดถ้ายังไม่ทำการทดสอบรันจริงในสภาพแวดล้อมที่เชื่อมต่อกันครบถ้วน หากระบบเชื่อมโยงระหว่าง Frontend และ Backend จะต้องจำลองการโยนไฟล์จริง ตรวจสอบสถานะการเชื่อมต่อ และยืนยันว่าไฟล์ไปถึงโฟลเดอร์ปลายทางบนเครื่องจริงของพี่เก่งอย่างสมบูรณ์แบบ การอ้างแค่ "โค้ดเสร็จแล้ว/บิวด์ผ่านแล้ว" แต่ไม่ทดสอบการไหลของข้อมูลจริง ถือว่าทำงานสะเพร่าและหยาบคายต่อกระบวนการทำงาน
- **Configuration Alignment Audit (ห้ามแยกตั้งค่า config แบบไร้การตรวจสอบผลกระทบข้ามระบบ):** ทุกครั้งที่เปลี่ยนหรือตั้งค่า Database ID, Storage Bucket, หรือ API Credentials ใดๆ ห้ามแก้ไขเฉพาะหน้าจุดใดจุดหนึ่งเด็ดขาด ต้องทำ Checklist ตรวจสอบทุกส่วนของระบบที่เชื่อมโยงกัน (React Config, Firebase console rules, local Python scripts) และทดสอบสิทธิ์การเข้าถึง (CORS, IAM rules) ควบคู่ไปด้วยเสมอเพื่อป้องกันการเกิดสถาปัตยกรรมแบบแยกส่วน (Split-Brain) และช่วยลดขั้นตอนการทำงานซ้ำซากลงเหลือศูนย์อย่างถาวร
- **Mandatory Audit Review & No Delegation (ส่งให้น้องออมตรวจทาน 100% ทุกครั้ง และห้ามใช้เขียนโค้ดแทน):** ทุกครั้งที่มีการแก้ไขหรือเขียนโค้ดจริง (HTML/CSS/JS หรือสคริปต์หลังบ้าน) ผมต้องส่งให้น้องออม (Aom) ตรวจทานความถูกต้อง ความปลอดภัย และความคลีนโดยใช้คู่มือและสแกนผ่านสกิล `scrutinize` และ `karpathy-guidelines` ทุกครั้งเสมอโดยไม่มีข้อยกเว้น และผมห้ามโยนงานเขียนโค้ดหรือมอบหมายงานพัฒนาระบบให้น้องออมทำแทนอย่างเด็ดขาด เนื่องจากน้องออมทำหน้าที่เป็นผู้ตรวจสอบโค้ดเท่านั้น ไม่มีบทบาทหรือสิทธิ์ในการเขียนโค้ดเข้าระบบ
- **Ponytail Coding Principle (Lazy Senior Coder):** บังคับใช้เกณฑ์กรองโค้ดของ Ponytail (YAGNI -> Stdlib -> Native -> Existing -> One-liner -> Minimal) ก่อนเริ่มลงมือเขียน ห้ามสร้าง Abstraction, Wrapper หรือ Helper ซับซ้อนโดยไม่จำเป็นเด็ดขาด เพื่อประหยัด Token และรักษาความเรียบง่ายของระบบ
- **HTML/SVG Performance Optimization & OBS Zoom Alignment Rules (กฎคุมประสิทธิภาพการเรนเดอร์ HTML/SVG และพิกัดซูม OBS):**
  - หลีกเลี่ยงการใช้ `will-change` หรือ `translateZ(0)` บน SVG paths (โดยเฉพาะ `stroke-dashoffset`) และ absolute overlays แบบพร่ำเพรื่อเพื่อเลี่ยงปัญหา Layer Explosion ที่ทำให้ GPU/CPU ทำงานหนักจนภาพกระตุก (Jank)
  - ห้ามใช้ `transition: transform` บน parent container ร่วมกับการตั้งค่า promotion layer (`will-change: transform` หรือ `translateZ(0)`) บน nested overlays เด็ดขาด เพื่อป้องกันตำแหน่ง overlays ขยับช้ากว่าพื้นหลังและเกิดการเหลื่อมหลุดจากจุดอ้างอิงขณะทำการซูม (Compositor Lagging)
  - คุมขอบเขตการอัปเดต UI บนกล่อง overlays ที่มีการเปลี่ยนแปลงข้อความหรือค่ากราฟย่อยๆ ด้วย `contain: strict;` เสมอ เพื่อจำกัดขอบเขต Style, Layout และ Paint ไม่ให้ทะลักและกระตุ้นการคำนวณใหม่ไปทั้งหน้าจอ
  - ห้ามใส่ข้อจำกัดขนาด `max-width: 100%` และ `max-height: 100%` บน container ที่ถูก scale ใน OBS Mode เพื่อป้องกันไม่ให้ขนาดกล่องหลักถูกเบี่ยงเบนตามขนาดบราวเซอร์ และคงสัดส่วนพิกัด canvas-to-SVG 1:1 ไว้ได้อย่างเที่ยงตรงในทุกระดับการซูม

## 🐞 บันทึกประเด็นด้านการเขียนโค้ด (Coding Issue Log)

### [Theme: Celestial Being]
- **ปัญหา HTML/JS (Theme Switcher refresh bug)**: หน้าเว็บสลับธีมค้างเมื่อเปลี่ยนกลับมาเป็นสว่าง (Light Mode) ในหน้าระบบคอนเทนต์ (`WTJ_Project/WTJ_Story/dashboard/index.html` และ `wtj_calendar_dashboard.html`)
  - *สาเหตุ*: CSS variables ใน `style.css` และ inline style ถูกประกาศและ override ไว้บน Selector `html.dark-theme` และ `html.color-theme` แต่ฟังก์ชันสลับธีม `setTheme` ใน JavaScript ทำการเขียนทับคลาสแค่บน `document.body` โดยไม่ได้อัปเดตคลาสของ `document.documentElement` (`<html>` element) ทำให้สไตล์สว่างที่สืบทอดจาก `:root` ไม่ได้รับการกู้คืนหลังสลับกลับมาจากสีเข้ม (ค้างที่สไตล์มืด)
  - *วิธีแก้ไข*: ปรับแก้ฟังก์ชัน `setTheme` ให้เปลี่ยนคลาสทั้ง `document.body.className` และ `document.documentElement.className` ไปพร้อมๆ กัน เพื่อให้ล้างคลาสสีเข้มออกตอนกลับมาเป็นสว่าง
- **ปัญหา WebKit CSS Variables rendering bug (Theme not changing immediately)**: เมื่อกดสลับธีมผ่านปุ่ม หน้าเว็บไม่ยอมอัปเดตสไตล์และสีตามธีมใหม่ทันที ต้องกด Refresh หน้าจอถึงจะหาย (พบบ่อยบน macOS/iOS ที่ใช้ Safari หรือ Chrome/WebKit Engine)
  - *สาเหตุ*: มีการประกาศและผูก CSS Variables ของธีมต่างๆ ไว้ที่ Selector `:root.theme-name` หรือ `html.theme-name` (เช่น `:root.dark-theme`) ซึ่งเบราว์เซอร์ตระกูล WebKit มีบั๊กที่จะไม่ยอมประมวลผลหรือคำนวณค่าตัวแปร CSS บนระดับ `:root`/`html` ใหม่ทันทีเมื่อมีการสลับคลาสผ่าน JavaScript แบบ dynamic

### [Feature: OBS Keyboard Zoom & Scale Persistence]
- **การอิมพลิเม้นท์ระบบ OBS Mode พร้อมการย่อ/ขยายด้วยคีย์บอร์ด**:
  - *รายละเอียดฟีเจอร์*: เมื่ออยู่ในโหมด OBS (ผ่านการเพิ่มคลาส `obs-mode` ใน `body`) พื้นหลังจะมืดสนิท ซ่อนเครื่องมือทั้งหมด และ Centering คอนโซลโฮโลแกรมหลัก ในโหมดนี้ผู้ใช้สามารถกดแป้นพิมพ์เพื่อซูมได้โดยตรง:
    - กดปุ่ม `+` หรือ `=` : ขยายขนาดแผงควบคุม (สูงสุด 2.5)
    - กดปุ่ม `-` หรือ `_` : ย่อขนาดแผงควบคุม (ต่ำสุด 0.4)
    - กดปุ่ม `0` : รีเซ็ตขนาดกลับเป็น 1.0 (ค่าเริ่มต้น)
  - *วิธีการเก็บค่าแบบถาวร (Persistence)*: เมื่อสเกลเปลี่ยนไปจะทำการเซฟลงใน `localStorage` ด้วยคีย์ `obsScale` ทันที และจะยิงเรียกอ่านตอนโหลดหน้าเว็บเพื่อส่งค่าไปที่ตัวแปร CSS Custom Property `--obs-scale` เพื่อให้สเกลคงอยู่ตลอดแม้ผู้ใช้จะรีเฟรชหน้าเว็บหรือเปิดใหม่
  - *สไตล์ CSS และ JS ที่ใช้*:
    ```css
    .visualizer-container {
      transform: scale(var(--obs-scale, 1.0));
      transform-origin: center center;
      transition: transform 0.1s ease-out;
    }
    ```
    ```javascript
    let obsScale = parseFloat(localStorage.getItem('obsScale')) || 1.0;
    
    function updateObsScale() {
      document.documentElement.style.setProperty('--obs-scale', obsScale);
      localStorage.setItem('obsScale', obsScale);
    }
    
    document.addEventListener('keydown', (e) => {
      if (document.body.classList.contains('obs-mode')) {
        if (e.key === '=' || e.key === '+') {
          obsScale = Math.min(2.5, obsScale + 0.05);
          updateObsScale();
        } else if (e.key === '-' || e.key === '_') {
          obsScale = Math.max(0.4, obsScale - 0.05);
          updateObsScale();
        } else if (e.key === '0') {
          obsScale = 1.0;
          updateObsScale();
        }
      }
    });
    ```

### [Feature: HTML Performance Optimization & OBS Zoom Alignment Fix (Round 8.8)]
- **ปัญหาวิดีโอกระตุก (Jank/FPS Drop) และตำแหน่ง HTML Overlay เหลื่อมหลุดเวลาซูมในโหมด OBS**:
  - *สาเหตุ*:
    1. **SVG will-change layer explosion**: การใช้ `will-change: stroke-dashoffset` บน path จำนวนมากใน SVG ทำให้เบราว์เซอร์พยายามสร้าง composite layers แยกบน GPU มากเกินไป (GPU promotion overload) ทั้งๆ ที่ GPU ไม่รองรับ hardware acceleration บน dashoffset ส่งผลให้ CPU ทำงานหนักและเฟรมเรตร่วง
    2. **Compositor Lagging**: การใช้ `transition: transform` บน parent container ร่วมกับการตั้งค่า `will-change: transform` และ `translateZ(0)` บน nested absolute overlays ทำให้ตำแหน่งของ HTML overlay ขยับช้ากว่าพื้นหลัง SVG ส่งผลให้เกิดการกระตุกและตำแหน่งเหลื่อมกันขณะเปลี่ยนอัตราซูม
    3. **CSS Layout Invalidation**: การอัปเดตข้อมูลข้อความและแถบสถานะใน overlay ถี่ๆ กระตุ้นการ recalculate layout และ repaint ไปทั้งหน้าจอ
    4. **OBS Layout Distortion**: การกำหนด `max-width: 100%` และ `max-height: 100%` บนกล่องที่ถูกสเกล (`.visualizer-container`) ทำให้ขนาดของ container ถูกบีบตามขนาดหน้าจอเบราว์เซอร์ของ OBS ที่ไม่ได้สเกลแบบ 1:1 ส่งผลให้พิกัด absolute ของ overlays เบี่ยงเบนและหลุดตำแหน่งเมื่อซูมในโหมด OBS
  - *วิธีแก้ไข*:
    1. **ปลด will-change ออกจาก SVG paths**: ลบคุณสมบัติ `will-change: stroke-dashoffset` ออกทั้งหมด เพื่อให้เบราว์เซอร์ยุบรวมการเรนเดอร์ลงในแผ่น raster เดียว ช่วยลดภาระการจัดสรรหน่วยความจำ GPU
    2. **ลบ transition และ GPU promotion บน overlays**: ลบ `transition: transform` และเอา `will-change` กับ `translateZ(0)` ออกจาก absolute overlays เพื่อให้ตำแหน่งของ overlays อัปเดตและถูกวาดซิงค์ไปพร้อมกับ parent container เสมอ
    3. **ตีกรอบ Layout ด้วย CSS Containment**: ใส่ `contain: strict;` บน overlays เพื่อบล็อกไม่ให้การคำนวณและวาดใหม่ (Style/Layout/Paint) ทะลุออกไปนอกพื้นที่ ช่วยเพิ่มความเร็วการเรนเดอร์
    4. **ลบ Scale Constraints**: เอา `max-width: 100%` และ `max-height: 100%` ออกจาก scaled container และปล่อยให้การย่อ/ขยายควบคุมผ่าน CSS transform สเกลพิกัด 1:1 เท่านั้น เพื่อล็อกให้ตำแหน่ง HTML overlays สอดคล้องกับพิกัด Canvas จริงในทุกระดับการซูม

    2. ใช้ `window.onerror` คอยดัก Error แล้วแสดง alert โชว์หน้าจอสำหรับ "ทุกๆ ข้อผิดพลาด" โดยไม่ได้กรองข้ามข้อผิดพลาดประเภท `Script error.` จากส่วนขยายภายนอก (Browser Extensions) ส่งผลให้เวลาผู้ใช้รันแอปออฟไลน์ผ่าน `file://` บราวเซอร์จะพ่นแจ้งเตือน JS Error ปลอมมาขัดจังหวะการทำงาน
  - *กฎเหล็ก:*
    1. สำหรับหน้าที่มีวัตถุขนาดคงที่ (เช่น หน้ากระดาษพรีวิว A4 `210mm` / `794px`) ต้องตั้งค่า Breakpoint ให้เหมาะสม (ล่าสุดปรับเป็น `1400px`) เพื่อให้ยุบแสดงผลแนวตั้งได้อย่างสบายตา และใช้การย่อ-ขยายแบบไดนามิกผ่าน `ResizeObserver` ร่วมกับสไตล์ CSS `zoom: var(--preview-scale-factor, 1)` แทนการฟิกซ์ `transform` รวมถึงต้องบังคับ `zoom: 1 !important` ใน `@media print` ด้วยทุกครั้งเพื่อให้หน้าตอนสั่งพิมพ์ไม่มีการลดสเกลค้าง
    2. เมื่อเขียน `window.onerror` ในแอปที่ต้องการให้ทำงานออฟไลน์ผ่าน `file://` ได้ จะต้องเขียนบล็อกตรวจสอบและละเว้น (Ignore) ข้อผิดพลาดที่เป็น `Script error.` หรือเกิดขึ้นที่พิกัดบรรทัด `0` เสมอ เนื่องจากบราวเซอร์มักพ่นค่าเหล่านี้ออกมาเมื่อส่วนขยายบราวเซอร์ภายนอกในเครื่องผู้ใช้เกิด Error ขึ้นมาเอง เพื่อไม่ให้หน้ากาก Error นั้นมารบกวนการใช้งานปกติ และให้หันไปครอบบล็อกการทำงานหลัก (เช่น DOMContentLoaded) ด้วย `try-catch` ท้องถิ่นแทนเพื่อสกัดหาความจริงตอนทดสอบระบบ
- **การล็อกข้อจำกัดฟิลด์ Credit Term และการปิดตัวแปร Mock Data (Credit Term Select & Mock Data Control - GHN168):**
  - *ข้อผิดพลาด:* 
    1. ปล่อยให้ฟิลด์เครดิตเทอม (Credit Term) เป็นช่องป้อนข้อความอิสระ ทำให้ผู้ใช้ป้อนข้อมูลปนตัวเลขและตัวอักษร ส่งผลให้สูตรคำนวณวันกำหนดชำระเงินพังบนตาราง Sheets
    2. ลืมควบคุมเงื่อนไขการสร้างข้อมูลจำลอง ทำให้ข้อมูล Mock Data เติมออโต้ทับข้อมูลจริงเวลาดีพลอยจริงเพื่อใช้งาน
  - *กฎเหล็ก:* 
    1. ต้องล็อกค่าเครดิตเทอมเป็นปุ่มเลือกดร็อปดาวน์ (Select Dropdown) เสมอ (เช่น 30 วัน, 45 วัน, 60 วัน หรือทันที) เพื่อป้องกันข้อผิดพลาดในการป้อนข้อมูลดิบ
    2. ต้องล็อกค่า Mock Data ผ่านตัวแปรควบคุม `ghn168_disable_mock` ใน `app.js` เสมอ เพื่อการตรวจเช็คและการทดสอบระบบจริงอย่างแม่นยำ
- **การซิงค์ Service Worker และ Cache Busting สำหรับแอปออฟไลน์ (Cache Busting for file:// Apps):**
  - *ข้อผิดพลาด:* ไม่ได้ทำการอัปเดต Service Worker หรือ Cache Busting หลังจากที่มีการแก้ไขโค้ด HTML/JS ทำให้บราวเซอร์ยังคงดึงไฟล์ cache เดิมมาแสดงผล ส่งผลให้ผู้ใช้ไม่เห็นฟีเจอร์ใหม่ที่เพิ่งดีพลอย
  - *กฎเหล็ก:* ทุกครั้งที่ปรับโค้ดและทดสอบแอปพลิเคชันที่ใช้ Service Worker ต้องอัปเดตเวอร์ชันของ Cache หรือตัว Service Worker (`sw.js`) เสมอเพื่อทำ Cache Busting เคลียร์ข้อมูลเก่าให้ผู้ใช้เห็นของใหม่ทันที
- **การพิมพ์กระดาษล้นหน้าและการปรับแต่งสัดส่วนลายเซ็นเฉพาะเอกสาร (PDF Print Overflow & Custom Signature Refinement):**
  - *ข้อผิดพลาด:* 
    1. ปล่อยให้ความสูงกระดาษปริ้นท์ถูกล็อกไว้ที่ `height: 276mm` และลืมปลด `min-height: 100vh` ของโครงสร้างหุ้ม (html, body, .app-container) ใน `@media print` ทำให้เวลาสั่งพิมพ์ใน Safari/บราวเซอร์ทั่วไป ตัวครอบจะล้นมาร์จินพิมพ์และพ่นหน้ากระดาษว่างเกินออกมาแผ่นที่ 2 เสมอ
    2. ลอจิกการดึงชื่อประเภทเอกสารสำหรับตั้งชื่อไฟล์แนะนำ PDF (`updateDynamicDocTitle()`) อ้างอิงจาก input radio `name="docType"` ที่ไม่มีอยู่จริง ทำให้ชื่อไฟล์ถูกแนะนำเป็น "เอกสารการเงิน" เสมอ และลืมใส่คำนำหน้า "Feltz_"
    3. ไม่ได้ซ่อนช่องลงลายเซ็นผู้รับเอกสาร (ฝั่งซ้าย) ของใบวางบิลและใบเสร็จตามเลเอาต์ใหม่ รวมถึงไม่ได้จัดเรียงช่องขวาชิดขวาสุด และกำหนดบทบาทเหนือ/ใต้เส้นลายเซ็นไม่ตรงเงื่อนไขของ Feltz Studio
  - *กฎเหล็ก:* 
    1. สำหรับสไตล์ปริ้นท์ (`@media print`) ห้ามล็อกความสูงขั้นต่ำหรือล็อกความสูงเต็มแผ่นบน parent container เด็ดขาด ต้องทำการ override `min-height: 0 !important; height: auto !important;` ให้กับ wrapper ทุกชั้น (`html`, `body`, `.app-container`, `.main-content`, `.view-panel`, และกล่องครอบกระดาษทั้งหมด) และตั้ง `.print-paper` ให้มี `height: auto !important; min-height: 0 !important; box-sizing: border-box !important;` เพื่อให้ความสูงหดสั้นรัดตามความสูงของเนื้อหาจริง ป้องกันปัญหาล้นหน้าว่างหน้าที่ 2 ตลอดกาล
    2. ฟังก์ชันอัปเดตหัวเรื่อง/ชื่อไฟล์เพื่อปริ้นท์ (`updateDynamicDocTitle()`) ต้องตรวจเช็คประเภทเอกสารจากตัวแปรสเตทปัจจุบันโดยตรง (เช่น `currentDocType`) และต้องเติมพรีฟิกซ์ตาม CI เช่น **`'Feltz_'`** ไว้ด้านหน้าเสมอ (เช่น `Feltz_ใบวางบิล_INVxxx`)
    3. สำหรับเอกสารทั้งหมดที่ไม่มีความจำเป็นต้องใช้ลายเซ็นลูกค้า (เช่น ใบวางบิล, ใบเสนอราคา, ใบเสร็จรับเงิน) ให้ซ่อนกล่องผู้รับเอกสารฝั่งซ้าย (`prevCustomerSigBox.style.display = 'none'`) และกำหนดจัดกล่องลายเซ็นขวาให้อยู่ชิดขวาเสมอ (`sigRow.style.justifyContent = 'flex-end'`) พร้อมจัดตำแหน่งเงื่อนไขหัวข้อลายเซ็นเหนือ/ใต้เส้นให้ตรงเป๊ะตามเงื่อนไขของบริษัท (เช่น ใบวางบิลไม่มีหัวข้อด้านบน แต่มี "ในนาม เจ้าของกิจการ Feltz Studio" ด้านล่าง)
    4. ต้องเปิดช่องหมายเหตุ (Remarks) ให้กรอกและพรีวิวได้ครบทุกประเภทเอกสาร โดยหากช่องว่างเปล่าให้ซ่อน แต่ถ้ามีข้อมูลให้พรีวิวขึ้นมาเพื่อความอเนกประสงค์
