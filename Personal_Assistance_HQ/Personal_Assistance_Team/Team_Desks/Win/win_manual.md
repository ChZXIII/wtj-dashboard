## 📝 บันทึกความจำเทคนิค (Technical Logs - พี่เฟิสสอนมา)
- **ตู้เก็บกุญแจและไฟล์สถานะ (Credentials & State Folder):** ไฟล์สิทธิ์ `token_sheets.json` และไฟล์เก็บสถานะเทรดคริปโต `trade_sync_state.json` ย้ายไปจัดระเบียบรวมไว้ที่โฟลเดอร์ **`credentials/`** ตรงรูทของโปรเจกต์ ChZ_Agent_Corp แล้วครับ สคริปต์ของผมถูกแก้ไขพาธเพื่อไปเรียกไฟล์จากโฟลเดอร์นี้เรียบร้อยครับ
- **Dashboard PnL Fix:** ถ้าเลข % ติดลบน้อยเกินไป (อย่างที่เคยเป็น -0.4x%) ให้เช็คว่าโค้ดใช้ `totalInvest` (ยอดรวมจริง) หรือ `tC_v` (ยอดเก่าที่ค้างอยู่) พี่เฟิสสอนว่าต้องใช้ยอดรวมจริงเสมอครับ!
- **History Backfill:** ถ้าต้องการดูข้อมูลย้อนหลังไปถึงต้นปี ผมมีสคริปต์ `backfill_history.py` ในโฟลเดอร์ scratch เป็นต้นแบบไว้ใช้คำนวณย้อนหลังได้ครับ
- **Clean Data & Row Management (อัปเดต 06/05/2026):**
  - **ห้ามเว้นช่องว่าง:** เวลาลงข้อมูลใน Sheet ห้ามปล่อยให้มีแถวว่างระหว่างข้อมูลเด็ดขาด ถ้ามีแถวว่างเหลืออยู่ผมต้องใช้แถวนั้นก่อน ไม่ใช่ไปแทรกแถวใหม่เพิ่มครับ
  - **Explicit Formatting:** เวลาลงวันที่ (Date) ต้องระบุ Format เป็น `dd/mm/yyyy` เสมอ ห้ามปล่อยให้ Google Sheets แปลงเป็นเลข Serial Number (เช่น 46148) เอง เพราะจะทำให้พี่เก่งอ่านไม่ออกครับ
- **Chronological Sync Logic (อัปเดต 12/05/2026):**
  - **Timeframe:** ห้ามอิงวันที่ปัจจุบัน (`today_ts_start`) แบบตายตัว เพราะจะพลาดออเดอร์ที่เกิดหลังจากที่ผมรันตอนเช้า! ผมต้องมองย้อนหลังไป **48 ชั่วโมง** เสมอ และจัดกลุ่มออเดอร์ตาม "วันที่ซื้อจริง" เพื่อถมช่องว่างให้ถูกต้อง
  - **Condition Routing (BTC/Mod):** ออเดอร์ซื้อ BTC ทุกไม้ต้องถูกแยกคัดกรอง "แบบรายไม้ (Per-order)" ไม่ใช่จับรวมกันก่อน:
    - ถ้ายอดซื้อไม้ไหน **<= 300 บาท** 👉 ให้ส่งไปจดที่ tab **`Mod`**
    - ถ้ายอดซื้อไม้ไหน **> 300 บาท** 👉 ให้ส่งไปจดที่ tab **`BTC`**
- **👑 กฎเหล็กและสถาปัตยกรรมชีต (อัปเดตล่าสุด 18/05/2026):**
  - ** Bitkub Scope Constraints:** ที่ Bitkub จะมีแค่ **เหรียญ KUB เท่านั้น** ที่จะถูกซิงค์เพื่อเขียนลงหน้าแท็บ **`Thyme`** (ห้ามยุ่งกับ BTC ใน Bitkub เด็ดขาด เพราะพี่เก่งไม่ซื้อ BTC ที่นี่)
  - **📐 BTC Header Row Offset Rule:** หน้าแท็บ `BTC` มี 2 Header rows ดังนั้น แถวเป้าหมายสำหรับเขียนข้อมูลของ **วันที่ N** ในเดือนนั้น จะต้องเป็น **`แถวที่ N + 2`** เสมอ! ห้ามแทรกแถวใหม่ และห้ามเขียนเหลื่อมทับวันอื่นเด็ดขาด!
  - **📈 Mod Dynamic SUM Range Rule:** สูตรผลรวมในช่อง `C30` และ `E30` ในแท็บ `Mod` ต้องอัปเดตสูตรเป็น `=SUM(C5:C29)` และ `=SUM(E5:E29)` (คลุมข้ามแถวว่างบัฟเฟอร์) เสมอ เพื่อให้เวลาผมแทรกแถวใหม่แล้วขอบเขตการบวกของ Google Sheets ขยายตามแบบอัตโนมัติ
  - **🎨 Thyme & Mod 5-Column & Formatting Schema:** หน้าแท็บ `Thyme` และ `Mod` มีโครงสร้าง **5 คอลัมน์** (วันที่, Week, ยอดเงิน, ราคา, จำนวนเหรียญ) แตกต่างจากเหรียญอื่นๆ ที่มี 4 คอลัมน์:
    - ผมต้องคำนวณเลขสัปดาห์แบบไดนามิกคือ **`last_week + 1`** เสมอ
    - ผมต้องสั่งดีไซน์จัดสี **ACCENT_BG & ROW_BG** รวมถึงเส้นขอบเข้มสีเขียวพรีเมียมให้ทุกช่องตารางเหมือนช่องอื่นๆ ทันทีที่แทรกแถวใหม่สำเร็จ (ห้ามเว้นช่องสีขาวเด็ดขาด!)
  - **⏰ Bitkub Servertime Sync Principle:** ห้ามคำนวณ Signature โดยใช้เวลาของเครื่องผมเด็ดขาดเพราะจะเกิดบั๊ก Clock Drift (ทำให้ติด Error 6 - Invalid Signature) ผมต้องดึงเวลาจากเซิร์ฟเวอร์โดยตรงผ่าน `/api/v3/servertime` มาสร้าง Signature เสมอครับ!

## 🧠 กฎเหล็กควบคุมการเขียนโค้ด (Development Principles - Karpathy's Rules)
เพื่อควบคุมไม่ให้โค้ดระบบหลังบ้านพังง่ายและซับซ้อนเกินไป ผมจะยึดแนวทางนี้ในการเขียนโค้ดและทำระบบซิงค์เสมอครับ:
- **Think Before Coding (คิดก่อนลงมือ):** ไม่เดาโครงสร้างชีตหรือเงื่อนไข API เองเด็ดขาด ถ้าจุดไหนคลุมเครือหรือไม่แน่ใจ ผมจะหยุดและถามพี่เฟิสเพื่อยืนยันก่อนลงมือทำ
- **Simplicity First (เรียบง่ายที่สุด):** เขียนสคริปต์ซิงค์ข้อมูลให้สั้น กระชับ และตรงจุดที่สุด ไม่เขียนโค้ดหรือสร้างโครงสร้างข้อมูลเผื่ออนาคตที่พี่เก่งไม่ได้สั่ง
- **Surgical Changes (แก้ไขเฉพาะจุดแบบผ่าตัด):** เวลาแก้ไขสคริปต์ (`master_sync_agent.py` หรือตัวอื่นๆ) เพื่อเพิ่มฟีเจอร์หรือแก้บั๊ก ผมจะแตะเฉพาะโค้ดจุดที่ต้องการแก้เท่านั้น ไม่ไปแก้อื่นๆ ฟอร์แมต หรือความเห็นข้างเคียงเด็ดขาด เพื่อป้องกัน Side Effects
- **Goal-Driven Execution (วัดผลด้วยความชัวร์):** ทุกครั้งที่ซิงค์ข้อมูลหรือรันสคริปต์แก้ชีต ต้องตรวจสอบผลลัพธ์ความถูกต้องหลังซิงค์เสมอ และถ้ามีโอกาสเสี่ยง ต้องหาวิธีทดสอบในชีตทดสอบ (Test Sheet) หรือสำรองข้อมูล (Backup) ก่อนลุยจริง
